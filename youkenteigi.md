# YouTubeコメント半自動返信システム 要件定義書

## 1. プロジェクト概要
YouTubeチャンネルの未返信コメントを取得してGemini(`gemini-2.5-flash`)で返信案を自動生成する機能と、その内容を人間が目視確認・修正した上でYouTubeに手動トリガーで投稿する機能を持つローカルシステムを構築する。
誤爆や不適切な返信、別人のコメントへの返信を防ぐため、システムは「取得・生成」と「確認・投稿」の処理を完全に分離し、必ず人間の目視確認を挟む設計とする。

## 2. システム構成
* **実行環境**: ローカルPC (Python 3.10以降)
* **UIフレームワーク**: Streamlit
* **データベース**: SQLite3 (標準ライブラリ。状態管理用)
* **外部API**: 
  * YouTube Data API v3 (コメントの取得・投稿、OAuth 2.0認証)
  * Google GenAI SDK (Gemini APIを利用した文章生成)

## 3. アプリケーションの実行フロー（2段階プロセス）

### フェーズ1：データ準備（バックグラウンド処理）
ユーザーがUI上の「コメント取得＆返信案生成」ボタンを押下すると以下の処理が実行され、完了後に待機状態（一時停止）となる。
1. **Fetch**: YouTube APIを実行し、未返信のコメント一覧を取得。
2. **Generate**: 取得した各コメントに対してGemini APIを実行し、返信案を生成。
3. **Save**: 「コメント主の名前」「元のコメント文」「生成された返信案」をSQLiteデータベースに `status = 'pending'`（未承認）として保存。

### フェーズ2：確認・修正・投稿（ヒューマンインザループ処理）
データベースに保存された内容をStreamlitの画面上に展開し、人間の操作を待つ。
1. **Review & Edit**: `status = 'pending'` のレコードを画面に一覧表示する。各行には「元のコメント」と「返信案（テキストエリア）」を表示し、ユーザーがその場で返信案を自由に書き換えられるようにする。
2. **Approve**: 確認・修正が完了したコメントに対して、ユーザーが「承認（Approve）」チェックボックスをオンにする。
3. **Post**: ユーザーが「選択したコメントを投稿」ボタンを押下した場合のみ、承認チェックが入っているコメントをYouTube API経由で実際に返信投稿する。投稿成功後、データベースのステータスを `status = 'posted'` に更新し、未処理リストから非表示にする。

## 4. データモデル設計
SQLite上に以下のテーブル `comments_state` を作成すること。
* `comment_id` (TEXT, PK): YouTubeのコメントID
* `video_id` (TEXT): 動画ID
* `author_name` (TEXT): 視聴者のアカウント名
* `original_text` (TEXT): 視聴者のコメント本文
* `generated_reply` (TEXT): 生成された返信テキスト（ユーザーによる直接編集が可能）
* `status` (TEXT): 現在の処理状態 (`pending` / `approved` / `posted` / `error`)

## 5. エラーハンドリングと安全制御
* **強制ストップ制御**: ユーザーの明示的な「投稿」ボタン押下アクションがない限り、YouTube APIの `comments.insert` メソッドは絶対に実行されない設計とすること。
* **ターゲット不整合防止**: 画面表示時、必ず「誰の（author_name）」「どのコメント（original_text）に対する返信か」を視覚的に隣接して表示し、ユーザーが対象者を誤認しないUI構造とすること。
* **システムインストラクション**: 返信生成時のプロンプト（Gemの指示文）はソースコードに直書きせず、外部ファイル（`.env` または `prompt.txt` 等）から動的に読み込むこと。
